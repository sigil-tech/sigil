# Sigil Constitution

## Core Principles

### I. Privacy-First

All raw telemetry (file paths, terminal commands, git metadata, process names, window
focus) MUST stay on the engineer's machine. Only summarized, user-reviewed context may
be sent to external LLMs via cloud inference. Fleet reporting sends only anonymized
aggregate counts and rates — never raw events, file paths, code content, or query text.

Raw event retention defaults to 90 days, configurable by the engineer. The engineer has
a kill switch (accessible via `sigilctl purge` and the Insights view) to wipe the local
store at any time. Data flows MUST be inspectable via the Insights view and
`sigilctl export`. Every line of code that touches workflow data is open-source and
auditable — "trust us" is not the privacy model; "read the code" is.

Privacy is the product's trust foundation. Violating it is an existential risk.

### II. Daemon-First

`sigild` is the core product and the first thing to validate. The observation →
analysis → suggestion loop must be useful on its own, without the shell or OS. New
capabilities start in the daemon and prove value before being surfaced through higher
layers (shell, OS).

The daemon MUST run stable for 48+ hours, under 50MB RSS, on constrained hardware
(2017 MacBook Pro, Intel, 8–16GB RAM). If a feature cannot be validated at the daemon
layer, it is not ready for the shell layer. The shell (Tauri + Rust + Preact) is a
surface for the daemon's intelligence, not the product itself.

### III. Interface-Driven Design

Consumers MUST depend on interfaces, not concrete types. Every cross-package boundary
uses an explicit Go interface (`store.EventReader`, `inference.Backend`,
`collector.Source`, `notifier.Platform`, `actuator.Actuator`). Mocks are generated by
`mockery` from these interfaces and committed to `internal/mock/`. This enables isolated
unit testing without real databases, real LLMs, or real OS services.

New packages MUST define or consume interfaces at their boundaries — never accept
`*ConcreteType` in a public API when an interface will do. Single-method interfaces
MUST use the `-er` suffix per Go convention (`EventReader`, `PatternWriter`). Interface
definitions live in the package that owns the abstraction, not the consuming package.

### IV. DAG Discipline

The package dependency graph MUST be a directed acyclic graph. `event` is the leaf
package with zero internal imports. No package may import a package that would create
a cycle. New packages MUST be placed in the DAG before implementation begins.

Dependency order:

```
event → config → store → inference → collector → notifier → analyzer → actuator → fleet → socket → cmd/sigild
```

The inference signal pipeline follows a three-layer architecture within this DAG:

1. **Signal layer** — tool-agnostic filesystem, VCS, and process events from `collector`
2. **Adapter layer** — optional tool-specific adapters that enrich raw signals
3. **Surface layer** — future plugins that expose enriched signals to external tools

Violations of the DAG are build-blocking defects.

### V. Inference Architecture

`sigild` owns its inference layer. A managed `llama-server` process (from `llama.cpp`)
runs quantized LLMs directly on the engineer's hardware. The daemon routes between
local and cloud models via a single `inference.Backend` interface with four explicit
routing modes:

| Mode | Behavior |
|------|----------|
| `local` | Always local; fail if unavailable |
| `localfirst` | Local, fall back to cloud |
| `remotefirst` | Cloud preferred, local fallback |
| `remote` | Always cloud |

The default mode is `localfirst`. The target is 80%+ of queries handled on-device.
The on-device model is LFM2-24B-A2B at Q4\_K\_M quantization (~2B active params via MoE,
<14GB on disk, <120ms p50 latency). The cloud frontier path is opt-in and previewed
before any context leaves the machine.

Swapping models or runtimes is a config change — not an API change. The `inference`
package depends on no specific runtime; it drives `llama-server` via its
OpenAI-compatible API endpoint.

### VI. Progressive Trust

The suggestion system earns the right to act through demonstrated utility. Five
discrete notification levels gate how aggressively the system surfaces suggestions:

| Level | Name | Behavior |
|-------|------|----------|
| 0 | Silent | No output |
| 1 | Digest | Periodic summaries only |
| 2 | Ambient | Passive suggestion bar, no interrupts |
| 3 | Conversational | Surfaced on request |
| 4 | Autonomous | Auto-execute with >90% confidence + reversible undo path |

Confidence scores increase with observation count — the daemon never shows a suggestion
until it has observed the pattern multiple times. The first week is the noisiest; by
week three, only suggestions the engineer acts on survive. At Level 4, every
auto-executed action MUST have a documented undo path before it may fire. Trust is
earned, never assumed.

### VII. Minimal Dependencies

The daemon uses pure Go with no CGo requirement. SQLite via `modernc.org/sqlite` (pure
Go, WAL mode, single writer). No web frameworks, no ORMs, no dependency injection
frameworks. The dependency surface stays tight:

- `fsnotify` — filesystem events
- `go-toml/v2` — config parsing
- `modernc.org/sqlite` — local store

New dependencies require explicit written justification before being added. The daemon
ships as a single statically-linked binary with fast startup and low memory footprint.

### VIII. Observable by Default

Every subsystem MUST be queryable. The Unix socket server (documented in
`docs/socket-protocol.md`) exposes JSON-over-socket methods for all daemon state.
`sigilctl` provides CLI access to events, suggestions, patterns, config, fleet preview,
and AI queries. All data stored in SQLite is exportable via `sigilctl export`.
Structured logging (`slog`) is used throughout with consistent key names.

The daemon's behavior MUST be understandable from its outputs alone — no black boxes.

### IX. Fleet as Enterprise Layer

The fleet layer is enterprise-only and explicitly opt-in per engineer. It deploys on
the organization's own infrastructure (Helm chart or NixOS module) — no data flows
through Sigil's servers. The Fleet Reporter sends only:

- Anonymized aggregate counts (query volumes, acceptance rates, routing ratios)
- Adoption tier classification (no individual attribution)
- Build velocity trends (correlated to tiers, not individuals)

The fleet layer NEVER receives raw events, file paths, code content, query text, or
anything that identifies a specific engineer's work. What the Fleet Reporter will send
is always visible in the Insights view before it leaves the machine.

## Go Code Standards

This section is non-negotiable. All Go code in this repository MUST conform to these
standards. Code review and CI enforce them. Shortcuts are not acceptable.

### Idiomatic and Effective Go

All Go code MUST follow [Effective Go](https://go.dev/doc/effective_go) as the
canonical authority for style and idiom decisions. This is not a suggestion — it is the
standard. Every author and every AI agent working on this codebase is required to
internalize it. When a style question arises, Effective Go answers it.

**Package design**

- Package names are lowercase, single-word, no underscores, no mixedCaps.
- Package names are the noun they represent (`store`, `collector`, `event`) — not
  generic containers (`util`, `common`, `helpers`).
- Exported identifiers do not stutter: `store.Store` is wrong; `store.DB` or
  `store.Client` is right.

**Naming**

- Acronyms are all-caps: `SQLiteStore`, `JSONEncoder`, `HTTPClient`.
- Unexported names use short, contextual names within their scope; exported names are
  descriptive without redundancy.
- Error variables are named `ErrXxx`; error types are named `XxxError`.
- Single-method interfaces use the `-er` suffix: `EventReader`, `PatternWriter`.

**Error handling**

- Every error is handled explicitly. `_ = f()` is forbidden unless there is a
  documented reason the error is provably irrelevant.
- Errors are wrapped with context: `fmt.Errorf("open store: %w", err)`. The wrapping
  message is lowercase, no trailing punctuation, and describes the operation — not the
  error.
- Sentinel errors (`var ErrNotFound = errors.New(...)`) are used for expected
  conditions callers must handle. Error types (structs implementing `error`) are used
  when callers need to inspect fields.
- `errors.Is` and `errors.As` are the only unwrapping mechanisms — never string
  matching on error messages.
- Errors are returned up the call stack and logged once at the boundary where they
  are handled, not at every layer.

**Concurrency**

- Goroutines are always named at their launch site with a comment explaining their
  lifetime.
- `context.Background()` is only used at goroutine roots with no parent context. All
  blocking operations accept and propagate `ctx`.
- `sync.WaitGroup` and `errgroup.Group` are the standard tools for goroutine
  coordination. Prefer `errgroup` when any goroutine may return an error.
- Channels communicate ownership transfer. Mutexes protect shared state. Do not mix
  idioms without justification.
- The SQLite store is single-writer: `store.Open` sets `SetMaxOpenConns(1)`. No
  concurrent writers may be added.

**Functions and methods**

- Functions do one thing. If a function needs a multi-paragraph comment to explain
  what it does, it should be split.
- Receiver names are one or two letters, consistent throughout all methods on a type
  (`s *Store`, not `store *Store` or `this *Store`).
- Named return values are used only when they materially aid clarity (e.g., multiple
  same-typed returns, or deferred assignments to named results). They are never used
  just to avoid declaring a variable.

**Structs and initialization**

- Struct fields are always named in composite literals. Positional literals are
  forbidden except for trivially obvious cases (e.g., `image.Point{1, 2}`).
- Zero values are meaningful by design wherever possible.

### Static Analysis and Formatting

- `gofmt` is canonical. No diff from `gofmt` output is acceptable.
- `go vet` must pass clean.
- `golangci-lint` runs in CI with at minimum: `errcheck`, `staticcheck`, `govet`,
  `revive`, `exhaustive`, `wrapcheck`, and `noctx`.
- `make check` (fmt + vet + lint + test) MUST pass before every commit.

### Testing

- Tests use table-driven patterns with `t.Run` subtests. Each subcase is independently
  named and independently runnable.
- Test names are `TestXxx/case_description` — lowercase with underscores for the
  subcase portion.
- Integration tests use real SQLite (in-memory via `openMemory(t)`), not mocks, for
  the store layer. Mocks are reserved for cross-package boundaries.
- Mocks are generated by `mockery` from interfaces. Hand-written mocks are forbidden.
- No test may call `time.Sleep` for synchronization. Use channels, `sync.WaitGroup`,
  or polling with `testutil.Eventually`.
- `t.Parallel()` is called in every test that can safely run in parallel.
- Coverage target: >50% enforced by `make coverage`, aspirational >90%.

## Technical Constraints

### Hardware Target

The reference platform is a 2017 MacBook Pro (Intel, 8–16GB RAM). If `sigild` runs
well on constrained hardware, it validates the cost story for enterprise deployment on
any fleet. Memory budgets: daemon ≤50MB RSS, local inference model via zero-copy mmap,
Sigil Shell (Tauri) ≤200MB.

### Protocol Stability

The Unix socket protocol is documented in `docs/socket-protocol.md`. The wire format
is newline-delimited JSON: one `Request` line, one `Response` line per connection.
Adding new methods requires only a `srv.Handle(...)` call — no protocol changes. The
protocol shape is a contract with the shell and CLI; breaking changes require a version
bump in the protocol document and a migration path.

### Concurrency Model

SQLite single-writer: `store.Open` sets `SetMaxOpenConns(1)`. Do not add concurrent
writers. Context propagation is mandatory — `context.Background()` is only used at
goroutine roots without a parent context. All blocking operations accept and respect
`ctx`.

## Development Workflow

### Commit Discipline

Each logical change is one commit. Phase work follows dependency order — never skip an
issue or batch commits across issues. Build verification
(`go build ./... && go vet ./... && go test ./...`) runs after every change. Commits
reference issue numbers when closing GitHub issues.

### No Dead Weight

No dead code — if it's unused, delete it. No duplication — shared helpers go in the
leaf package (`event`) or the consuming package. No over-engineering — three similar
lines are better than a premature abstraction.

### Dependency Justification

Every new external dependency requires a written justification (one sentence minimum)
in the PR description explaining: what problem it solves, why the standard library
alone is insufficient, and what the maintenance risk is.
